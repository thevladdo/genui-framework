# GenUI Tenant Isolation Statement

**Audience**: the security team of a customer running a GenUI deployment.

**Scope**: one GenUI deployment (one backend, one Redis, one Qdrant) serving multiple tenants, the customer's access architecture (e.g. insurance: `agente` vs `assicurato`; enterprise: `dipendente` vs `cliente`).

**Claim**: no request authenticated as tenant A can read, write, delete or influence data belonging to tenant B.

**As of**: 2026-07-20. Line references are to this revision of the codebase; each row names the symbol so the reference can be re-verified with `grep` after updates.

## The one rule everything derives from

**The tenant of a request is resolved server-side from the presented API key, never from anything the client controls.** Keys are mapped to tenants in server config (`CLIENT_API_KEYS` / `ADMIN_API_KEYS`, entries `key:tenant`). No request body, header or query field can name a tenant.

- Key → tenant resolution: `KeyRegistry.authenticate` (`backend/auth/keys.py:96-110`)
- Enforced on every route as a FastAPI dependency: `require_client` / `require_admin` (`backend/auth/dependencies.py:90-127`)
- **Fail-closed**: with no keys configured, privileged routes refuse (403). The only override is the explicit dev flag `GENUI_DEV_OPEN=1` (`open_mode_context`, `backend/auth/identity.py:164-178`), which must never be set in a customer deployment; `deploy/smoke.sh` fails if it is.

Every store below then namespaces its data by that resolved tenant. Isolation is enforced at the storage key / query filter, not by route-level discipline: a new route reusing these stores inherits the boundary.

## What tenants DO share (read this before signing)

Honesty about the boundary's nature: inside one deployment, isolation is **logical, not physical**.

- **The process.** All tenants are served by the same Python workers. There is no per-tenant memory partition; isolation relies on the key namespacing and filters above.
- **Redis and Qdrant instances.** Separation is by key prefix (Redis) and mandatory payload filter (Qdrant), not separate databases. Nothing except the backend can reach them (the deploy compose publishes neither).
- **The LLM engine and its key (BYOK, per-deployment).** Tenants share the engine; the per-tenant budget separates spend, and prompts for shared renders are built from segment archetypes, not raw user data.
- **The operator.** Whoever administers the VM sees everything. This document makes no claim against the deployment's own operator.

If two audiences must not share even these (separate failure domain, separate engine key, physical separation): run **two deployments**. The model is one-command-per-deployment precisely so that this remains cheap (`deploy/README.md` §Bring-up).

## Failure posture

- **No keys configured** → privileged routes fail closed (403), they do not degrade to open. (`backend/auth/identity.py:164-178`)
- **Redis unavailable** → stores fall back to bounded per-process memory and reconnect with backoff. The fallback uses the same tenant-namespaced keys, so isolation properties are preserved; what degrades is durability and cross-worker consistency, visibly (`GET /health` → `"redis": "reconnecting"`, status `degraded`). (`backend/utils/redis_conn.py`,
  `tests/test_redis_reconnect.py`)
- **Embedding/LLM misconfiguration** → loud startup or request errors, never a silent fallback to a different provider (a typo must not send documents to a third party). (`backend/llm/embeddings.py`, `tests/test_embeddings.py`)
